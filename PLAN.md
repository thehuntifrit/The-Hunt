# 実装計画書 (IMPLEMENTATION PLAN) - 最終決定版

## 1. 修正の目的
「3秒周期の監視」という誤った実装を完全に排除し、本来の「1分監視・3秒描画」仕様を復元した上で、描画リソースの最適化と復帰時の即時反映を実現します。

## 2. 実装の要件（正しき仕様）

### A. 監視と判定の分離（監視階層）
- **1分周期監視 (Tier B)**: 
    - ランクタブ・拡張タブで**選択されている全モブ**を対象に実行。
    - ステータス判定、湧き判定、通知処理を行う。
    - **重要**: タブがバックグラウンドにあっても、精度維持のため**論理監視は継続**する。
- **3秒周期描画 (Tier C)**:
    - **動的昇格のみ**: 湧き境界まで「1分（リアル時間）以内」のモブ、または「活動中（なう）」のモブのみを 3秒周期の描画対象とする。
    - それ以外のモブは、原則 1分周期（Tier B）の更新に統合し、無駄な描画を排除する。

### B. 表示ガード (Visibility Guard) と復帰動作
- **非表示時 (Background)**:
    - **DOM描画のみを完全に停止**する。
    - バックグラウンド中の 1分周期の論理監視は継続し、最新ステータスは保持し続ける。
- **復帰時 (Foreground)**:
    - ブラウザタブが表示状態に戻った瞬間、**即座に全件の論理再計算および描画更新（force update）を実行**する。
    - これにより、待機時間なしで最新の状態を画面に反映させる。

### C. 計算の最適化
- 特殊条件（天候・月齢）の探索は、討伐報告や再読込時などの「トリガー発生時」のみ実施。
- 定期更新ループ内では、その計算済み時刻との比較（O(1)）のみを行い、重い検索は絶対にループさせない。

## 3. 修正対象ファイル
- js/app.js: 動的 Tier 管理と、可視性変化時の即時再計算ロジック。
- js/cal.js: 周期ループからの冗長な再検索を物理的にブロックするガードの追加。

## 4. 期待される効果
- バックグラウンド時の CPU 負荷を 0.1% 未満に。
- 画面復帰時の遅延ゼロと、最新情報の確実な反映。
