# The Hunt - アプリケーション仕様書 (AI管理者向け)

> **最終更新日: 2026-04-12**

このドキュメントは、本アプリケーションの内部構造、データフロー、およびロジックの定義を詳細に記述したものです。
AIがシステムを理解し、保守・拡張を行うための公式仕様書として機能します。

## 1. システム概要

FF14のモブハント（S級、A級モブ、特殊FATE）の出現時間を管理・共有するためのプログレッシブ・ウェブ・アプリ (PWA) です。Firebaseをバックエンドに使用し、リアルタイムでの討伐時間、メモ、マップ情報を同期します。

## 2. アーキテクチャ構成

Vanilla JS (ES Modules) をベースとしたシングルページアプリケーション (SPA) です。

### ディレクトリ構造

- `/js`: コアロジック (UI、データ管理、計算、サーバー通信)
- `/css`: スタイル定義 (Vanilla CSS)
- `/json`: 静的データ定義 (モブ基礎データ、座標、メンテ情報)
- `/maps`: エリアマップ画像
- `/sound`: 通知音。

## 3. データモデルと管理 (`dataManager.js`)

### State オブジェクト

アプリケーションの全状態を管理します。

- `userId`, `lodestoneId`: 認証情報。
- `mobs`: 計算済みのモブデータの配列。
- `filter`: ランク、エリア選択、表示ステップの状態。
- `maintenance`: サーバーメンテナンス情報。

### データの永続化

- **LocalStorage**: ユーザーID、認証状態、フィルター設定、通知設定を保持。
- **IndexedDB (`HuntDB`)**: 重いデータ（モブデータ、座標キャッシュ、出現計算キャッシュ）を保持。

### 3.3 JSONデータ構造定義

AIがデータを更新・追加する際のスキーマ定義です。

#### 3.3.1 モブ基礎データ (`mob_data.json`)

- **JSONルート**: `{ "mobs": { "ID": { ... } } }`
- **主要フィールド**:
  - `rank` (string): S, A, F
  - `name` (string): モブ名
  - `area` (string): エリア名（`mob_locations.json` のキーと一致）
  - `repopSeconds` (number): 基礎リポップ秒数（最短）
  - `maxRepopSeconds` (number): 基礎リポップ秒数（最大）
- **出現条件フィールド** (任意):
  - `timeRange`: `{ "start": 0, "end": 23 }`
  - `weatherSeedRange`: `[min, max]` (0-99)
  - `weatherDuration`: `{ "minutes": 1400 }`
  - `moonPhase`: "満月", "新月" など
  - `conditions`: 特定条件下でのみ有効なサブ条件。

#### 3.3.2 地点・座標データ (`mob_locations.json`)

- **JSONルート**: `{ "エリア名": { ... } }`
- **フィールド**:
  - `mapImage` (string): `/maps/` 配下のファイル名
  - `locations` (array):
    - `id` (string): 地点ID（例: "CW_101"）
    - `x`, `y` (number): 0-100 の相対座標
    - `mob_ranks` (array): その地点に出現可能性のあるランク（例: `["S", "A"]`）

## 4. 時間計算ロジック (`cal.js` & `worker.js`)

> [!IMPORTANT]
> **cal.js は「変更禁止 (Frozen Module)」です。**
> 時間換算、天候計算、特殊条件探索のアルゴリズムは完成されており、外部からの修正は一切認められません。計算ロジックに関する調整が必要な場合は、呼び出し側の引数またはデータ定義 (json) の変更で対応してください。

システムの核心となる時間計算は、現実時間 (Unix Time) からエオルゼア時間 (ET) を算出し、各種条件をシミュレートすることで行われます。

### 4.1 エオルゼア時間と天候予測

- **時間換算**: 現実の175秒がETの1時間に相当します (`ET_HOUR_SEC = 175`)。
- **天候サイクル**: ET 8時間（現実の1400秒）ごとに天候が変化します。
- **天候シード値**: Unix Timeから算出された `eorzeanDays * 100 + timeChunk` をベースに、ビット演算を用いて 0-99 のシード値を生成します。
  - 計算式: `((seed << 11) ^ seed) >>> 8) ^ seed) % 100` （概略）
  - `mob_data.json` 内の `weatherSeedRange` と照合して条件判定を行います。

### 4.2 特殊出現条件の探索 (`findNextSpawn`)

以下の3つの条件を順次重ね合わせ（Intersect）、最初に出現可能な期間を特定します。

1. **月齢条件**: `getEorzeaMoonInfo` により、32 ET日周期の月相を計算。「新月 (phase 32.5-4.5)」「満月 (phase 16.5-20.5)」などの期間を抽出。
2. **天候条件**: 指定された天候シードが連続して必要な時間（`weatherDuration`）維持される期間を探索。
3. **時刻条件**: ET 0:00-23:59 の範囲で指定された `timeRange` を判定。

### 4.3 リポップ（再出現）タイマー計算

討伐報告 (`last_kill_time`) に基づき、以下の優先順位で `minRepop` / `maxRepop` を決定します。

1. **通常時**: `last_kill_time` + 基礎リポップ秒。
2. **サーバー復旧直後**: `last_kill_time` がサーバー復旧時刻 (`serverUp`) 以前の場合、以下の短縮ルールが適用されます。
   - **S/Aランク**: `(serverUp + repopSeconds * 0.6)`
   - **FATE (ランクF)**: `(serverUp + repopSeconds)` （短縮なし）
3. **ステータス判定**:
   - `Next`: `now < minRepop`
   - `PopWindow`: `minRepop <= now < maxRepop`
   - `MaxOver`: `now >= maxRepop`
   - `ConditionActive`: `PopWindow` 以降 かつ 特殊条件が現在合致している。

### 4.4 並列処理 (`Worker`)

これらの計算（特に20日先までの全探索）はメインスレッドをブロックしないよう `Web Worker` で実行されます。結果は `IndexedDB` にキャッシュされ、同一条件での再計算を回避します。

## 5. UIコンポーネントと設計 (`mobCard.js`, `sidebar.js`, `app.js`)

### 5.1 レイアウト構造

- **PC画面 (1024px以上)**:
  - **サイドバー**: 左端固定。最小幅 `56px`、ホバー展開時 `216px`。
  - **メインリスト**: 中央配置、固定幅 `480px`。
  - **詳細パネル (`mobcard-pane`)**: 右端 `sticky` 配置。リストと同一画面内に非スクロールで表示。
- **モバイル画面 (1023px以下)**:
  - 上部ヘッダー（時計表示等）および下部ナビゲーションを固定（各 `40px`）。
  - 詳細表示はフルスクリーンのオーバーレイ（`mobcard-overlay`）として表示。

### 5.2 状態表示と動的ソート

- **優先順位**: `MaxOver` > `PopWindow` > `Next` > `Maintenance` の順。
- **ゲージ**: `minRepop` から `maxRepop` にかけて色が遷移し、100%（最大）を超えると表記が `超過` に変化。

### 5.3 ネットワーク状態と同期

- **3-Tierガバナンス**: 第9項に詳述される通り、モブの緊急性と「現在選択されているタブ」に基づき、計算・更新頻度を3段階（1分/ET1分/停止）で動的に制御し、不要な計算を徹底的に排除する。
- **計算結果のダイレクト同期**: 1分（およびET1分）ごとの正規計算結果（`repopInfo`）とUIを完全に同期させ、表示の正確性を保証する。
- **インクリメンタルDOM更新**: 第8.4項に記述される通り、ステータス遷移時のみ要素を抽出し、`insertBefore` 等を用いて目標グループの適切な位置へ直接差し込むことで渲染負荷を最小化する。
- **楽観的UI更新**: マップチェック等の操作時、サーバーからのレスポンスを待たずにDOMとStateを先行更新する。

## 8. UI/UX 設計ガイドライン

指示に基づいた継続的なデザイン整合性を保つための定義です。

### 8.1 Z-Index 階層 (0-100)

| 階層名 | 値 | 対象要素 |
| :--- | :--- | :--- |
| **Loading** | 100 | 初期読み込みオーバーレイ |
| **Overlay/Toast** | 96-98 | 通知、拡大鏡、ツールチップ、ヘッダー |
| **Modal** | 90 | 討伐報告、認証モーダル |
| **Modal Backdrop** | 80 | モーダル背面の半透明マスク |
| **Mobile Overlay** | 60 | モバイル個別の詳細画面 |
| **Sidebar/Sticky** | 40 | サイドバー、PC詳細パネル |

### 8.2 視覚エフェクト

- **白光 (`Condition Active`)**: 特殊条件合致時または窓内に、`moblist-highlight-white` 等の専用クラスを付与。低負荷を優先した静的な強調（`box-shadow` および `border` の強調）を行う。
- **プログレスバー**: 左から右へ `transform: scaleX()` が実行される。CSS変数 `--color-progress-*` に従い色が決定される。

### 8.3 デバイス別インタラクション・プロトコル

- **PC**:
  - ホバー: リスト項目のハイライト、サイドバー展開。
  - 右クリック: マップ上のマグニファイア（拡大鏡）レイヤーの起動。
- **モバイル**:
  - タップ: 項目の選択。
  - ダブルタップ（1秒以内）: マップ座標のチェック（済/未）切り替え。
- **スクロール制御**: オーバーレイ表示時は `body-lock` クラスにより背面のスクロールを禁止（`overscroll-behavior: contain`）。

## 6. サーバー連携 (`server.js`)

### Firebase (v12.4.0)

  - `mob_status`: 討伐時間の共有。
  - `mob_locations`: マップ上のピン状態の共有。

Workers プロキシ経由で Lodestone サイトをスクレイピングし、自己紹介欄の検証コードを照合してキャラクター所有権を確認します。

## 9. パフォーマンスとCPU最適化 (`app.js`, `mobSorter.js`)

アプリケーションのCPU負荷を最小限に抑えつつ、正確なリアルタイム監視を実現するために、以下の「3階層計算ガバナンス」を採用しています。

### 9.1 更新フェーズ (3-Tier Governance)

現在のユーザーフィルタ（選択中のランクタブ等）およびモブの状態に基づき、計算・同期頻度を3段階に制御します。

| フェーズ | 周期 | 対象・挙動 | 目的 |
| :--- | :--- | :--- | :--- |
| **Tier A** | 随時 | **非表示設定**（フィルタ外）のモブ。 | 計算、ソート、DOM更新を100%停止する。 |
| **Tier B** | 60秒周期 | **選択中のタブ内**で、残り時間が **1分以上** の全モブ。 | 全項目の計算結果（`repopInfo`）とUI表示を完全に同期させる。 |
| **Tier C** | 2.917秒周期 | **選択中のタブ内**で、残り（または超過）時間が **1分未満** の全モブ。 | 画面外であっても最短周期で状態変化を監視し、表示の変化があれば描画を行う。 |

- **Tier C の適用範囲**: 描画（DOM書き換え）そのものは可視領域での変化が優先されますが、状態監視とタイマーの同期は「選択タブ内」であれば全件継続されます。これにより、スクロール時に常に最新の時間が表示されます。

### 9.2 重い計算の実行抑制 (Cache-First Algorithm)

定期ループ（Tier B/C）内での負荷を最小化するため、天候・月齢を含むフル検索（`calculateRepop`）の実行を制限します。

1. **実行トリガー**: 起動時、討伐報告受信時、および現在時刻が「境界時刻（`nextBoundarySec`）」を追い越した瞬間にのみフル計算を実行。
2. **定常ルール**: 1分（またはET1分）ごとの計算周期において、正規の計算値（真値）を直接UIに反映させます。UI側で独自に時刻の引き算（概算）を行うことは禁止されています。

### 9.3 ソートと描画の最適化

1. **表示変化ガード**: 表示（分単位：H:MM）やステータスに変化がない限り、DOMの書き換えをスキップします。
2. **ソートの限定**: 進捗%が変動する `PopWindow` / 条件合致グループを除き、`NEXT` や `MAINTENANCE` グループの定常的なソートは停止し、順位を固定します。
3. **インクリメンタル更新**: 次項のプロトコルに従い、要素の差し替えを最小化します。

---

## 8.4 インクリメンタルDOM更新プロトコル

UIのチラつきを抑え、Reflow負荷を最小化するための規定です。

1. ステータスが遷移したモブ、または討伐報告されたモブのみをDOMから抽出します。
2. 目標グループ内のソート順位を特定し、`insertBefore` 等を用いて対象要素のみを直接差し込みます。他の安定している項目のDOMには一切関与しません。

## 7. 管理・運用上の注意点

- **コードの分離**: UI生成 (`mobCard.js`) とロジック (`cal.js`)、通信 (`server.js`) は完全に分離されています。
- **仕様の主権**: **この `specification.md` の記述が、本プロジェクトにおける唯一の「真実」であり、他の古いコメントや外部ドキュメント（削除済みを含む）よりも常に優先されます。**
- **クリーンコード**: JSDocや不必要なコメントを排除し、自己説明的な関数・変数名を維持してください。
- **パフォーマンス遵守**: いかなるUIの変更も、第9項に記載された CPU 最適化指針を逸脱してはなりません。
